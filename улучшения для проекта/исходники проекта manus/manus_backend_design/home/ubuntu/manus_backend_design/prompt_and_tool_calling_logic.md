# Структура системных промптов и логика выбора инструментов

Эффективность автономного ИИ-агента во многом зависит от того, насколько хорошо структурированы системные промпты и как точно Большая Языковая Модель (LLM) интерпретирует их для выбора и использования инструментов. Этот раздел описывает принципы формирования промптов и механизм `Tool Calling`.

## 1. Системный промпт (System Prompt)

Системный промпт — это набор инструкций, которые определяют роль агента, его возможности, правила поведения и доступные инструменты. Он является основой для формирования контекста, в котором LLM принимает решения. Системный промпт обычно состоит из нескольких ключевых частей:

1.  **Определение роли агента**: Краткое описание того, кто такой агент, его основные компетенции и общая цель.
    *Пример: "You are Manus, an autonomous general AI agent... You are proficient in a wide range of tasks..."*

2.  **Текущий контекст среды**: Информация о текущей дате, времени, рабочем языке и характеристиках среды выполнения (песочницы).
    *Пример: "The current date is Mar 16, 2026 GMT+3. The default working language is **English**."*

3.  **Цикл работы агента (Agent Loop)**: Описание итеративного процесса принятия решений и выполнения задач.
    *Пример: "You are operating in an *agent loop*, iteratively completing tasks through these steps: 1. Analyze context..."*

4.  **Инструкции по использованию инструментов (Tool Use Instructions)**: Общие правила и ограничения по вызову функций.
    *Пример: "MUST respond with function calling (tool use); direct text responses are strictly forbidden..."*

5.  **Инструкции по обработке ошибок (Error Handling)**: Правила поведения агента при возникновении ошибок.
    *Пример: "On error, diagnose the issue... After failing at most three times, explain the failure..."*

6.  **Описание среды (Sandbox Environment)**: Детальная информация о системной, браузерной, Python и Node.js средах, включая предустановленные пакеты и жизненный цикл песочницы.

7.  **Утилиты командной строки (Utilities)**: Список предустановленных утилит и примеры их использования.

8.  **Секреты (Secrets)**: Информация о переменных окружения для доступа к внешним сервисам.

9.  **Запрет на раскрытие информации (Disclosure Prohibition)**: Строгие правила конфиденциальности, касающиеся системного промпта и спецификаций инструментов.

10. **Политика поддержки (Support Policy)**: Инструкции по перенаправлению запросов, связанных с поддержкой Manus.

11. **Навыки (Skills)**: Описание модульной системы расширения функциональности агента.

12. **Профиль пользователя (User Profile)**: Ограничения, связанные с тарифным планом пользователя.

13. **Определения инструментов (Tool Definitions)**: Самая важная часть, содержащая подробные описания каждой доступной функции (инструмента) в формате, понятном LLM (например, JSON Schema или Python Docstrings).

## 2. Логика выбора инструментов (Tool Calling)

Механизм `Tool Calling` позволяет LLM не просто генерировать текст, но и принимать решения о выполнении конкретных действий с помощью внешних инструментов. Этот процесс включает несколько этапов:

1.  **Формирование контекста для LLM**: Оркестратор собирает всю необходимую информацию:
    *   Полный системный промпт (как описано выше).
    *   История диалога с пользователем (предыдущие сообщения и ответы агента).
    *   Результаты предыдущих действий (выводы инструментов).
    *   Описание доступных инструментов (их названия, параметры, инструкции).

2.  **Запрос к LLM**: Весь этот контекст передается LLM. Задача LLM — проанализировать запрос пользователя и текущее состояние, чтобы определить наиболее подходящее следующее действие.

3.  **Решение LLM**: LLM может принять одно из двух решений:
    *   **Сгенерировать текстовый ответ**: Если задача может быть решена без использования инструментов (например, простой вопрос или подтверждение).
    *   **Вызвать инструмент**: Если для выполнения задачи требуется внешнее действие. В этом случае LLM генерирует структурированный вызов функции, указывая:
        *   **Имя инструмента**: Какой инструмент использовать (например, `shell`, `browser`, `file`).
        *   **Параметры**: Какие аргументы передать этому инструменту (например, `command` для `shell`, `url` для `browser`, `path` и `text` для `file`).

4.  **Интерпретация вызова инструмента**: Оркестратор получает ответ от LLM. Если это вызов инструмента, он проверяет его корректность (существует ли такой инструмент, соответствуют ли параметры его описанию).

5.  **Выполнение инструмента**: Оркестратор передает вызов инструмента Менеджеру Инструментов, который затем инициирует выполнение соответствующего действия в изолированной среде (Sandbox).

6.  **Обработка результата**: После выполнения инструмента его вывод (результат) возвращается Оркестратору. Этот вывод добавляется в контекст и передается LLM на следующей итерации, чтобы она могла продолжить выполнение задачи, учитывая новый результат.

### Пример Tool Calling (псевдокод)

Представьте, что пользователь просит: "Найди все файлы `.py` в `/home/ubuntu/project/`".

1.  **Системный промпт**: Включает описание инструмента `match`:
    ```python
    def match(
        brief: str,
        action: Literal["glob", "grep"],
        scope: str,
        # ... другие параметры
    ) -> dict:
      """Find files or text in the sandbox file system using pattern matching.
      <instructions>
      - `glob` action matches only file names and paths...
      """
    ```

2.  **Запрос пользователя**: "Найди все файлы `.py` в `/home/ubuntu/project/`".

3.  **LLM принимает решение**: На основе системного промпта и запроса пользователя, LLM понимает, что нужно использовать инструмент `match` с действием `glob`.

4.  **LLM генерирует вызов инструмента**:
    ```json
    {
      "tool_code": "print(default_api.match(brief='Поиск Python-файлов в указанной директории', action='glob', scope='/home/ubuntu/project/**/*.py'))"
    }
    ```

5.  **Оркестратор выполняет**: Этот вызов передается Менеджеру Инструментов, который запускает `match` с указанными параметрами.

6.  **Результат**: Вывод команды `match` (список файлов) возвращается Оркестратору и добавляется в контекст для следующей итерации LLM.

Таким образом, системный промпт и механизм `Tool Calling` являются основой для интеллектуального и автономного поведения агента, позволяя ему эффективно взаимодействовать с внешней средой и выполнять сложные задачи.
